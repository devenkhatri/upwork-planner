The Solution

A decision cockpit for Upwork applications, not another job list.

The app ingests scraped Upwork listings, enriches each listing with AI strategy fields, saves the records in NocoDB, and serves a focused dashboard for deciding what to apply to, skip, archive, or inspect.

19scraped jobs in the included dataset

200records fetched per dashboard call before client-side filtering

How It Works

One listing becomes a decision-ready opportunity record.

This walkthrough follows the real ingestion flow in `scripts/setup-nocodb.mjs`: dataset row, AI enrichment, weighted score, database insert, then dashboard review.

1
Dataset

A scraped Upwork job enters the system.

The included dataset contains 19 records with title, description, budget, client history, proposals, location, tags, and screening questions.

Behind the scenes

The setup script reads `dataset_upwork-job-scraper_2026-05-24_06-55-21-455.json` from the project root.

2
AI Strategy

The listing is translated into strategy fields.

OpenRouter is asked to classify phase, complexity, risk flags, hooks, required questions, and proposal recommendations.

Behind the scenes

The prompt asks for strict JSON, which keeps the response usable as database columns instead of free-form notes.

3
Score

The system calculates a single apply score.

The score weights skill fit most heavily, then client quality, reward versus effort, and competition advantage.

Behind the scenes

The code computes: 0.4 x skill + 0.3 x client + 0.2 x reward + 0.1 x competition.

4
NocoDB

The enriched job is saved as a permanent record.

NocoDB becomes the searchable source of truth for job metadata, AI scoring, decision status, and archive state.

Behind the scenes

The script creates the `UpworkJobs` table if needed, then inserts every enriched record.

5
Dashboard

The freelancer works from a ranked pipeline.

The Next app fetches records, filters active versus archived jobs, sorts by score or client quality, and opens detailed proposal guidance.

Behind the scenes

Status changes are written back through `/api/jobs` so archived jobs stop distorting active dashboard stats.

function computeApplyScore(ai) {
  const score =
    0.4 * clamp(ai.skillFit, 0, 10) +
    0.3 * clamp(ai.clientQuality, 0, 10) +
    0.2 * clamp(ai.rewardVsEffort, 0, 10) +
    0.1 * clamp(ai.competitionAdvantage, 0, 10);
  return Math.round(score * 100) / 100;
}
1Starts the final scoring calculation for one job.
2-6Turns four buyer-relevant judgments into one comparable number.
3Skill fit gets the most weight because a fast win matters most.
4Client quality protects against low-value or unreliable buyers.
5-6Reward and competition keep the score grounded in market reality.
7The final score is rounded so it reads cleanly in the dashboard.
export async function updateJobsStatus(
  jobIds: string[],
  status: 'active' | 'archive'
): Promise<{ success: number; failed: number }> {
  const results = await Promise.allSettled(
    jobIds.map((id) => updateJobStatus(id, status))
  );
  const success = results.filter((r) => r.status === 'fulfilled' && r.value).length;
  return { success, failed: jobIds.length - success };
}

5-7Attempts every requested job update, even if one update fails.
8Counts how many jobs were successfully moved.
9Reports success and failure separately, so the UI can be honest.
export async function GET() {

  }
}
